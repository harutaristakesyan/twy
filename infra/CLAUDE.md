# infra/ — SST infrastructure modules

Single-source-of-truth for every AWS resource the twy app provisions. Imported lazily inside `sst.config.ts → run()` so SST can resolve order on its own.

> Read root `CLAUDE.md` first.

## Files

| File | Owns |
|---|---|
| `domain.ts` | Per-stage domain config (primary + aliases) and the `stageConfig()` selector. |
| `database.ts` | Aurora Serverless v2 (Postgres) cluster (`sst.aws.Aurora`) with `dataApi: true`, plus the small `sst.aws.Vpc` it needs. Per-stage scaling: prod `min: 0.5 ACU`, dev `min: 0 ACU` (auto-pause). |
| `storage.ts` | Files S3 bucket with multi-origin CORS. |
| `email.ts` | SES email identity per apex domain (auto-DKIM/SPF/DMARC via `sst.aws.dns()`). |
| `auth.ts` | Cognito user pool + app client + post-confirmation Lambda trigger. |
| `api.ts` | ApiGatewayV2 + Cognito JWT authorizer + per-route `sst.aws.Function` instantiation driven by `routes.ts`. |
| `web.ts` | `sst.aws.Router` (multi-domain CDN) + `sst.aws.StaticSite` (Vite SPA) + `/api/*` proxy to the API. |
| `routes.ts` | Single typed route table; `authRoutes` (public Cognito flows) + `appRoutes` (JWT-protected). |

## Conventions

- **Every module exports a factory function** (`createX(args) { ... }`) — never a top-level `new sst.aws.*` outside `sst.config.ts → run()`. This keeps ordering explicit and side-effects testable.
- **`/// <reference path="../.sst/platform/config.d.ts" />`** at the top of every module. The ambient types (`sst`, `aws`, `$app`, `$interpolate`, `$config`) come from there. Until `sst install` (or first `sst dev` / `sst deploy`) runs, your editor will flag these as unresolved — that's expected.
- **Cross-component access is via factory args**, not module-level globals. `createApi({ auth, db, filesBucket })` makes the dependency graph readable and prevents accidental ordering bugs.
- **No SSM**, no CloudFormation exports, no env-var injection helpers. Use `link[]` on Functions and `Resource.X` in handlers.

## Adding a new component

1. New file at `infra/<thing>.ts`.
2. `/// <reference path="../.sst/platform/config.d.ts" />` first line.
3. `export function create<Thing>(args: { ... }) { ... return { thing }; }`.
4. In `sst.config.ts → run()`: `const { create<Thing> } = await import("./infra/<thing>");` and pass it whatever upstream resources it depends on.
5. If the new resource needs to be reachable from a Lambda handler, add it as a key in the `linkRegistry` in `infra/api.ts` and reference it from `routes.ts` via a new `LinkKey`.

## Adding a new HTTP route

1. Author the handler in `packages/functions/src/api/auth/` (public Cognito flows) or `packages/functions/src/api/<domain>/` (JWT-protected business logic).
2. Append a `RouteDef` to `infra/routes.ts` — `authRoutes` for public, `appRoutes` for JWT.
3. Set `linkKeys` to the resources the handler needs. The IAM permissions are derived; you don't write any policy.
4. `pnpm sst deploy --stage dev` (or push and let CI deploy).

## Pitfalls

- **`Cannot find namespace 'sst'`** — your editor hasn't seen `.sst/platform/config.d.ts` yet. Run `pnpm sst dev --stage <username>` once (or any `sst deploy`) to generate it.
- **`Property 'X' does not exist on type 'Resource'`** — same root cause; `sst-env.d.ts` is generated by SST after the first `sst dev` / `sst deploy` against the stage you're targeting.
- **A new alias domain fails DNS validation** — the Route53 hosted zone must already exist in the right AWS account before `sst deploy`. SST's `dns: sst.aws.dns()` adapter looks zones up; it doesn't create them.
- **Renaming `primaryDomain`** in `domain.ts` mid-flight forces SST to replace the underlying resources rather than update them. Always extend via `aliases`.
- **Cert is silently us-east-1** — SST handles this for `sst.aws.Router`, but if you spin up an `sst.aws.Cdn` directly, remember CloudFront refuses certs from any other region.
