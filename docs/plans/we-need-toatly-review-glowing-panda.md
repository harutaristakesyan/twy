# Branch ↔ User Ownership Refactor

## Context

Today, "the owner of a branch" is implemented inversely through `users.branch` (the user's own membership pointer), with no actual `ownerId` column on the branch table. The repository's `assignOwner()` helper enforces a single-owner-per-branch invariant by **clearing `users.branch = NULL`** for every other user on that branch when a new owner is set — so assigning multiple users to a branch via the user form silently breaks every time the branch's owner is updated.

We want:
1. A real, **optional** `ownerId` column on the `branch` table (FK → `users.id`, `ON DELETE SET NULL`).
2. **Branch creation**: `owner` is optional (currently required by Zod + UI rule).
3. **User creation**: `branch` is optional (currently required by UI rule only — Zod + DB are already permissive).
4. The branch's owner is shown in the UI (already exists in the list; preserved).
5. **Per user decision**: when an owner is assigned/changed on a branch, that user is automatically also set as a member (`users.branch = ownerId`), but other members keep their assignments. No peer-clearing.
6. **No backfill** — all existing branches start with `ownerId = NULL`; the inverse `users.branch` data is preserved untouched.

The user's `branch` column stays as a separate, independent membership pointer (many-to-one), orthogonal to `branch.ownerId`.

## Files to change

### Database

- `packages/db/src/schema/branch.ts` — add `ownerId: uuid().references(() => users.id, { onDelete: "set null" })` (use plain `() => users.id`; non-self-referential FKs don't need the `AnyPgColumn` cast). Re-export of `BranchRow`/`NewBranch` updates automatically via `$inferSelect/$inferInsert`.
- `packages/db/drizzle/0002_branch_owner_id.sql` — generated by `pnpm sst shell --stage <yourname> -- pnpm --filter @twy/db db:generate`. Expected SQL:
  ```sql
  ALTER TABLE "branch" ADD COLUMN "owner_id" uuid;
  --> statement-breakpoint
  ALTER TABLE "branch" ADD CONSTRAINT "branch_owner_id_users_id_fk"
    FOREIGN KEY ("owner_id") REFERENCES "public"."users"("id")
    ON DELETE set null ON UPDATE no action;
  ```
  No backfill UPDATE — all existing branches stay `NULL`.
- `packages/db/drizzle/meta/_journal.json` + `meta/0002_snapshot.json` — auto-generated; commit as-is.

### Core domain (`@twy/core`)

- `packages/core/src/branch/repository.ts` — significant rework:
  - **Drop** the `assignOwner()` helper entirely. Its peer-clearing semantics are the bug we're removing.
  - **`NewBranchInput`**: `ownerId` becomes `string | null | undefined` (optional).
  - **`UpdateBranchInput`**: `ownerId` already optional/nullable — keep, but the implementation changes.
  - **`createBranch`**: in a single transaction:
    1. If `ownerId` provided, call `ensureOwnerExists(tx, ownerId)`.
    2. Insert into `branch` with `id`, `name`, `contact`, `ownerId` (or NULL).
    3. If `ownerId` provided, `UPDATE users SET branch = <newBranchId> WHERE id = ownerId` (auto-membership). Do **not** touch any other user.
  - **`updateBranch`**: when `ownerId` key is present in payload:
    - If new value is `null`: just write `branch.ownerId = NULL`. Do **not** touch `users.branch` (the previous owner stays a member; admin can clear via user form if desired).
    - If new value is a uuid: validate via `ensureOwnerExists`, write `branch.ownerId`, and `UPDATE users SET branch = branchId WHERE id = newOwnerId` (auto-membership of the new owner).
  - **`listBranches`**: change the LEFT JOIN from `users.branch = branch.id` to `users.id = branch.ownerId`. The owner shape returned (`{id, firstName, lastName, email}`) does not change.
  - **`deleteBranch`**: stop manually NULLing `users.branch` (the existing FK on `users.branch` is already `ON DELETE SET NULL`, and the new `branch.ownerId` FK is too — Postgres handles both). Reduces the function to a single DELETE.
  - **`ensureOwnerExists`**: keep as-is; still validates the user row exists.

- `packages/core/src/branch/request.ts`:
  - `CreateBranchPayloadSchema.body.owner` → `z.uuid().nullable().optional()` (drop the `≥1 length` string check; switch to a real uuid validation since the field is now an FK uuid not free text).
  - `UpdateBranchPayloadSchema.body.owner` already accepts `string | null` — tighten to `z.uuid().nullable().optional()` for consistency.

- `packages/core/src/branch/response.ts` — no shape changes (`BranchResponse.owner` is already nullable).
